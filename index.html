Controle de versões

São sistemas que registra as mudanças feitas em um arquivo ou um conjunto de arquivos ao longo do tempo, de forma que você possa recuperar versões específicas.

Sistemas de controle de versões locais

Um dos métodos preferidos da maioria das pessoas quando vão desenvolver um projeto é criar um diretório e a locar o seu código dentro, essa prática e dotada por muitos, pois é simples por outro lado é suscetível a erros. É fácil esquecermos em qual diretório você está e acidentalmente incrementar uma funcionalidade no arquivo errado ou sobrescrever arquivos sem querer. Para lidar com essa problemática, desenvolvedores desenvolveram ferramentas capazes de resolver esse conflitos com sistemas que armazena uma maquina local arquivos que pode ser modificados sobre controle de versão. O RCS armazena as diferenças entre cada versão de um arquivo, e permite que seja retornados as versões anteriores.

Sistemas de Controle de Versão Centralizados

Outra situação que os desenvolvedores se depara e quando sua profissão exigem que haja trabalho em equipe para utilizar sistemas diferentes como resolver esses conflitos. Para lidar com isso, foram desenvolvidos Sistemas de Controle de Versão Centralizados (Centralized Version Control System ou CVCS). Esses sistemas, como por exemplo, o CVS, Subversion e Perforce, possuem um único servidor central que contém todos os arquivos versionados e vários clientes que podem resgatar (check out) os arquivos do servidor. Por muitos anos, esse foi o modelo padrão para controle de versão. Essa ferramenta oferece uma grande vantagem, especialmente sobre VCSs locais. Por exemplo, todo mundo pode ter conhecimento razoável sobre o que os outros desenvolvedores estão fazendo no projeto. Administradores têm controle específico sobre quem faz o quê; sem falar que é bem mais fácil administrar um CVCS do que lidar com bancos de dados locais em cada cliente. Entretanto, esse arranjo também possui grandes desvantagens. O mais óbvio é que o servidor central é um ponto único de falha. Se o servidor ficar fora do ar por uma hora, ninguém pode trabalhar em conjunto ou salvar novas versões dos arquivos durante esse período. Se o disco do servidor do banco de dados for corrompido e não existir um backup adequado, perdem-se tudo — todo o histórico de mudanças no projeto, exceto pelas únicas cópias que os desenvolvedores possuem em suas máquinas locais. VCSs locais também sofrem desse problema sempre que se tem o histórico em um único local, corre-se o risco de perder tudo.

Sistemas de Controle de Versão Distribuídos

São sistemas que surgiram para resolver essa problemática dos sistemas centralizados (CVSs). A criação do sistema de controle distribuído permite que o desenvolvedor não só faça copias das ultimas versões dos arquivos mas de todo o repositório e assim, se o servidor falhar qualquer um dos repositórios dos desenvolvedores pode ser copiados de volta para o servidor para ser restaurado. Exemplos desses sistemas ( git,mercurial, Bazaar ou Darcs).
